## 1. 文档目标与建设原则

本文件用于把开题阶段的研究方向收敛为可执行的产品需求，重点解决四个问题：系统到底服务谁、第一阶段必须做什么、哪些能力属于“可配置”而不是“硬编码”、以及怎样定义一个可验收的最小闭环。

- 坚持“先做成一个订单场景可运行产品，再抽象平台能力”的原则。
- 低代码能力优先服务订单业务，不追求第一阶段成为通用型零代码平台。
- 规则编排优先实现同步决策和自动处理，不引入复杂审批流或异步工作流引擎。
- 页面、字段、校验、显示逻辑和规则定义均采用结构化元数据保存。
- 所有关键业务动作必须可追踪、可审计、可回放。
- 所有开放性设计在本文件中做产品化收敛，后续以此作为设计与开发基线。

## 2. 项目定位与边界

系统定位为“面向中小电商的低代码订单管理系统”。它不是通用 ERP，也不是纯表单搭建平台，而是围绕订单履约流程，提供一套可被业务人员和实施人员共同使用的后台系统。

### 2.1 本期建设目标

- 实现一个可工作的系统原型，可供商家或测试用户实际操作。
- 支持订单全生命周期的基础管理：导入或创建、审核、锁单、分仓、发货、取消、完成。
- 支持通过元数据配置实体字段、列表页、表单页、详情页和操作按钮。
- 支持通过可视化规则编排实现审核规则、分仓规则和标签/备注自动化。
- 完成可部署、可演示、可记录日志的系统，为后续案例验证提供基础。

### 2.2 非目标范围

- 不在第一阶段实现完整 ERP、财务核算、供应链计划和 BI 大屏。
- 不在第一阶段建设复杂 BPMN 审批平台、任务中心和消息中心。
- 不在第一阶段大规模接入外部电商平台、OMS、WMS、支付系统，只预留集成接口。
- 不在第一阶段实现复杂多租户计费、租户自助注册和 SaaS 商业化能力。
- 不允许在规则引擎中执行任意自定义脚本，本期仅支持白名单节点与动作。

### 2.3 本期产品化收敛假设

由于原始需求只有大体方向，本期对关键模糊项做如下收敛，以便系统能够落地：

- 订单来源初期支持后台手工录入、Excel/CSV 导入和内部接口写入，不强依赖外部电商平台。
- 库存管理不是本期重点，分仓仅基于仓库可用性、区域和优先级做决策，不做完整库存扣减闭环。
- 低代码平台优先支持“订单主实体 + 订单明细 + 客户 + 仓库”周边配置，不做任意页面类型和任意业务引擎。
- 规则编排本质上是“业务决策流”，不是“人机协同审批流”；规则执行以同步处理为主。
- 平台先支持单商家或轻租户模式，但数据模型和权限设计需为后续隔离预留扩展位。

## 3. 目标用户与使用场景

| 角色 | 核心诉求 | 典型操作 | 本期重点支持 |
| --- | --- | --- | --- |
| 系统管理员 | 管理账号、权限、配置发布 | 创建用户、授权角色、维护字典、查看审计 | 是 |
| 订单运营 | 提高订单处理效率 | 查询订单、审核、备注、改仓、发货、取消 | 是 |
| 客服/审核人员 | 识别风险与异常订单 | 查看风险标签、人工审核、锁单、驳回 | 是 |
| 规则配置人员 | 减少重复人工判断 | 配置审核规则、分仓规则、自动标签规则 | 是 |
| 实施/开发人员 | 快速适配商家差异 | 配置字段、页面、扩展实体、导入模板 | 是 |
| 仓配人员 | 关心发货与物流信息 | 查看待发货订单、录入物流单号、批量发货 | P1 |

典型使用场景包括：商家把新订单导入系统后，系统自动执行审核规则并打标签；人工只处理需要复核的异常订单；审核通过后系统根据地址和仓优先级自动分仓；运营确认后批量发货；实施人员在不改代码的前提下，为某类商家增加“平台店铺”“买家留言”“客服备注”字段，并把这些字段配置到订单列表和详情页中。

## 4. 业务流程总览

### 4.1 订单全生命周期主流程

订单进入系统后进入“待审核”；系统执行审核规则，将订单流转为“自动通过”“人工审核”或“直接拦截”；通过审核的订单进入“待分仓”；系统根据收货区域、仓优先级、仓状态等条件自动或手动分仓；分仓完成后进入“待发货”；录入物流单号并执行发货动作后进入“已发货”；后续根据签收、人工确认等动作进入“已签收”或“已完成”；未发货订单可在符合条件时取消；发货后若发生售后则进入“售后中”。

- 输入：订单基础信息、商品明细、客户信息、地址信息、支付信息、订单来源信息。
- 处理：状态流转、审核判断、仓库匹配、标签打标、备注写入、物流信息绑定。
- 输出：订单状态、规则命中记录、业务日志、审计日志、统计指标。

### 4.2 本期重点业务链路

本期必须打通以下三条闭环链路：

1. 订单创建/导入 -> 自动审核 -> 人工复核 -> 审核通过/驳回。
2. 审核通过 -> 自动分仓/手工改仓 -> 待发货 -> 发货完成。
3. 配置新增字段 -> 发布页面配置 -> 页面自动渲染 -> 规则可引用新字段。

## 5. 详细需求说明

### 5.1 功能模块总表

| 模块 | 子模块 | 关键能力 | 优先级 |
| --- | --- | --- | --- |
| 用户与权限 | 登录、角色、菜单、操作鉴权 | 账号登录、角色授权、菜单控制、按钮控制、接口鉴权 | P0 |
| 订单管理后台 | 列表、详情、状态流转、批量处理 | 查询、审核、锁单、改仓、发货、取消、日志追踪 | P0 |
| 低代码配置平台 | 实体、字段、页面、动作配置 | 字段建模、页面渲染、配置发布、版本回滚 | P0 |
| 规则编排引擎 | 审核、分仓、标签、备注自动化 | 可视化编排、调试试跑、优先级执行、日志留痕 | P0 |

| 第三方集成 | 物流、支付、电商平台 | API 预留、Webhook、导入导出 | P2 |

### 5.2 订单管理后台

#### 5.2.1 模块目标

订单管理后台不是简单的订单列表，而是订单履约操作中台。本模块需要同时满足“高效处理”“过程可追溯”“可被规则驱动”三类目标。

#### 5.2.2 订单状态模型

订单采用“主状态 + 控制标记 + 业务子状态”的方式建模，避免所有含义都塞进单一状态字段。

| 维度 | 建议值 | 说明 |
| --- | --- | --- |
| 主状态 | 待审核、人工审核、待分仓、待发货、已发货、已签收、已完成、已取消、售后中 | 驱动主流程展示与可执行动作 |
| 控制标记 | 是否锁单、是否异常、是否命中规则、是否人工介入 | 用于界面标识和操作约束 |
| 业务子状态 | 支付状态、分仓状态、物流状态、退款状态 | 用于细化业务判断与筛选 |

状态流转约束如下：

- 新创建或导入订单默认进入“待审核”。
- 审核规则命中自动通过时，订单直接进入“待分仓”。
- 审核规则命中人工复核时，订单进入“人工审核”。
- 分仓成功后进入“待发货”，发货成功后进入“已发货”。
- 已发货订单不可取消，只能进入售后类流程。
- 锁单订单禁止执行发货、取消、改仓等破坏性动作，除非先解锁。

#### 5.2.3 订单列表页需求

订单列表页是运营人员最常使用的工作台，必须强调检索效率、批量处理能力和状态识别能力。

- 支持按订单号、外部订单号、客户姓名、手机号、收件地区、状态、支付状态、分仓状态、物流状态、仓库、标签、来源渠道、时间范围进行筛选。
- 支持按 SKU、商品名称、买家留言、客服备注进行模糊搜索。
- 支持保存常用筛选方案，例如“待审核订单”“今日待发货”“异常订单”。
- 支持显示关键列：订单号、来源、客户、金额、商品摘要、状态、仓库、标签、创建时间、最后处理时间。
- 支持列显隐、列顺序调整、列宽配置和状态色标展示。
- 支持批量选择并执行批量审核、批量打标签、批量导出、批量发货。
- 支持行内快速操作，但高风险操作必须二次确认并填写原因。
- 支持根据权限控制不同角色可见列和可执行操作。

#### 5.2.4 订单详情页需求

订单详情页应采用“总览 + 分区卡片 + 时间线/日志”的结构，至少包含以下区域：

- 订单概览：订单号、来源渠道、主状态、风险标签、金额摘要、仓库信息。
- 收货与客户信息：收件人、手机号、地址、客户等级、历史订单数。
- 商品明细：SKU、名称、规格、数量、单价、小计、备注。
- 金额信息：商品金额、优惠金额、运费、应付金额、实付金额。
- 履约信息：分仓结果、物流公司、物流单号、发货时间、签收时间。
- 规则命中记录：命中的规则名称、节点路径、动作结果、执行耗时。
- 处理日志与审计日志：谁在什么时间执行了什么动作，动作前后结果是什么。
- 扩展字段区：由低代码配置新增的业务字段，按配置自动渲染。

#### 5.2.5 订单操作与业务约束

| 操作 | 适用状态 | 关键校验 | 结果 |
| --- | --- | --- | --- |
| 审核通过 | 待审核、人工审核 | 非锁单，必要字段完整 | 进入待分仓 |
| 审核驳回 | 待审核、人工审核 | 需填写驳回原因 | 进入已取消并记录驳回原因 |
| 锁单/解锁 | 非已完成、非已取消 | 需记录原因 | 更新控制标记 |
| 自动分仓 | 待分仓 | 地址、仓规则、仓状态有效 | 写入分仓结果并进入待发货 |
| 手工改仓 | 待分仓、待发货 | 有权限且未发货 | 覆盖仓库并记录原因 |
| 发货 | 待发货 | 物流公司、物流单号完整 | 进入已发货 |
| 取消订单 | 待审核、人工审核、待分仓、待发货 | 未发货且未完成 | 进入已取消 |
| 标记异常 | 任意未完成状态 | 填写异常类型 | 写入异常标签 |
| 完成订单 | 已签收或人工确认场景 | 有权限 | 进入已完成 |

额外约束：

- 所有状态变更必须记录操作者、时间、原因、变更前值、变更后值。
- 人工操作应覆盖规则结果，但必须保留“人工覆盖”标记。
- 发货、取消、改仓、审核驳回属于高风险动作，必须进入审计日志。
- 批量操作应支持成功/失败明细回显，不能因为单条失败导致全部静默失败。

#### 5.2.6 批量处理与导入导出

- 支持订单模板导入，导入时执行字段校验、重复单识别和错误提示。
- 支持订单列表导出，导出字段可配置，受权限控制。
- 支持批量审核、批量打标签、批量发货，要求可查看处理结果汇总。
- 批量发货支持“导入物流单号文件”与“手工填写统一物流公司”两种模式。

#### 5.2.7 异常场景处理

本期需明确支持以下异常场景：

- 地址缺失或格式异常：进入人工审核或标记异常。
- 命中高风险规则：自动锁单并进入人工审核。
- 分仓失败：保持待分仓状态，记录失败原因，允许手工改仓。
- 发货失败：保留待发货状态，记录错误并允许重试。
- 已发布规则变更后导致处理结果差异：订单详情需能看到命中的规则版本。

#### 5.2.8 模块验收标准

- 运营人员可在列表页完成日常查询、筛选、批量操作。
- 单笔订单详情可以完整追踪状态流转、规则命中和人工处理记录。
- 系统至少支持一条“自动审核 + 自动分仓 + 发货”的订单闭环。
- 系统至少支持一条“命中异常规则 -> 人工审核 -> 驳回/通过”的订单闭环。

### 5.3 低代码配置平台

#### 5.3.1 模块定位

低代码配置平台的目标不是让用户任意造系统，而是在订单管理范围内，允许实施人员通过配置完成字段扩展、页面调整和操作入口控制。它必须做到“可配置、可发布、可回滚、可治理”。

#### 5.3.2 配置对象范围

| 配置对象 | 作用 | 本期要求 |
| --- | --- | --- |
| 实体定义 | 定义业务对象及其基础属性 | P0 |
| 字段定义 | 定义字段类型、校验、默认值、展示属性 | P0 |
| 字典/枚举 | 支持状态、标签、选项配置 | P0 |
| 页面 Schema | 定义列表、表单、详情页布局与组件 | P0 |
| 页面动作 | 定义按钮、批量操作、显示条件 | P0 |
| 视图权限 | 控制角色可见字段与可用动作 | P1 |
| 模板与复制 | 复制已有实体/页面配置作为模板 | P1 |

本期明确不支持：

- 任意拖拽生成门户站点或营销页面。
- 任意自定义 SQL、脚本代码或组件源码注入。
- 无约束地修改系统内置核心字段的存储语义。

#### 5.3.3 实体建模需求

平台应支持两类实体：

- 系统内置实体：订单、订单明细、客户、仓库、规则日志、审计日志等，允许扩展字段但不允许删除核心字段。
- 业务扩展实体：由实施人员新增，用于承载轻量辅助业务数据，可被订单表单或详情页引用。

实体配置至少包括：

- 实体名称、实体编码、描述、是否启用。
- 主显示字段、主键策略、是否支持列表页、是否支持详情页。
- 与订单的关系类型，例如一对一附属信息、一对多子表。
- 是否允许作为搜索项、是否允许作为规则输入源。

#### 5.3.4 字段配置需求

字段配置是低代码平台的核心，至少需要支持以下能力：

- 字段基础属性：名称、编码、类型、描述、默认值、是否必填、是否唯一。
- 存储属性：是否持久化、映射列名、是否允许为空、是否支持数组或 JSON。
- 展示属性：显示名称、提示文案、占位符、宽度、排序顺序、分组归属。
- 行为属性：是否可编辑、是否只读、是否可搜索、是否可排序、是否列表展示、是否详情展示。
- 校验属性：长度范围、数值范围、正则校验、枚举校验、自定义错误提示。
- 联动属性：显示条件、编辑条件、依赖字段、字典来源、关联实体来源。

本期建议支持的字段类型包括：文本、长文本、数字、金额、布尔、日期、日期时间、枚举、标签多选、手机号、地址、图片/附件引用、关联实体、子表。

#### 5.3.5 页面配置需求

页面配置至少覆盖列表页、详情页、编辑页三类视图。

列表页配置要求：

- 配置搜索区域字段、搜索组件、默认值和折叠策略。
- 配置表格列、格式化方式、状态标签、金额格式、时间格式。
- 配置行操作和批量操作的显示条件、权限码和确认文案。
- 支持不同角色显示不同列或不同操作入口。

编辑页配置要求：

- 支持按分组或卡片组织字段。
- 支持必填、只读、隐藏、条件显示等规则。
- 支持子表编辑，用于订单明细或扩展子实体。
- 支持基础组件映射，不要求实现自由布局，但需支持双列或单列布局。

详情页配置要求：

- 支持概要区、基础信息区、金额信息区、履约信息区、日志区。
- 支持时间线、标签区和子表格。
- 支持规则命中卡片和审计日志卡片作为标准组件复用。

#### 5.3.6 配置运行机制

- 所有页面配置保存为结构化 Schema，由运行时渲染引擎解析。
- 渲染引擎负责将字段类型映射到标准组件，例如枚举映射为下拉、金额映射为数字输入与金额展示。
- 页面运行时需同时结合字段配置、权限配置和当前订单状态决定最终可见内容。
- 扩展字段一旦发布，订单页面、导入导出、规则输入上下文应能够按配置自动感知。

#### 5.3.7 配置发布与版本治理

为避免配置失控，平台必须具备发布治理能力。

- 配置状态至少包括：草稿、已发布、已停用。
- 支持预览：在不影响正式页面的前提下查看渲染结果。
- 支持版本记录：每次发布都生成版本号、发布人、发布时间、备注。
- 支持回滚：可回退到历史发布版本。
- 已发布字段若被页面、规则、导入模板引用，不允许直接删除，应先提示依赖关系。
- 核心字段的编码、主数据类型发布后不允许随意变更，避免破坏数据兼容性。

#### 5.3.8 配置治理约束

低代码配置需要边界，避免平台变成不可维护的“万能配置器”。

- 核心订单主流程按钮由系统保留，不允许完全删除，但可按权限和条件控制显示。
- 字段编码必须全局唯一，发布后默认不可修改。
- 同一字段的展示规则、校验规则和规则引擎引用需要使用统一字段编码。
- 配置平台必须提供依赖分析，至少能提示“该字段被哪些页面/规则引用”。
- 配置错误必须在保存时校验，避免运行时才发现页面无法渲染。

#### 5.3.9 模块验收标准

- 不改代码即可新增订单扩展字段，并在列表页、详情页、编辑页展示。
- 新增字段可以被搜索、导出和规则编排引用。
- 页面配置支持预览、发布、回滚。
- 已发布配置发生错误时，系统能提示依赖与回滚入口，而不是直接破坏线上页面。

### 5.4 规则编排引擎

#### 5.4.1 模块定位

规则编排引擎用于把人工判断转化为可重复执行的业务决策流。它不是通用流程审批系统，而是围绕订单场景提供“条件判断 + 动作执行 + 结果留痕”的可视化规则能力。

#### 5.4.2 规则触发场景

| 场景 | 触发时机 | 典型用途 | 优先级 |
| --- | --- | --- | --- |
| 订单创建/导入后 | 新订单进入系统时 | 自动审核、打标签、锁单 | P0 |
| 审核通过后 | 准备进入履约环节时 | 自动分仓、设置优先级 | P0 |
| 发货前 | 待发货阶段 | 校验物流必填项、补充备注 | P1 |
| 人工重跑规则 | 运营手工触发时 | 修正配置后重新评估 | P1 |
| 外部事件触发 | 第三方回调或内部事件 | 预留后续扩展 | P2 |

#### 5.4.3 规则类型与执行策略

本期建议先落地三类规则：

- 审核规则：决定自动通过、人工审核、锁单、驳回。
- 分仓规则：决定分配仓库或给出候选仓建议。
- 标签/备注规则：自动写入标签、备注、优先级。

执行策略要求：

- 规则按场景归类，每个场景下允许存在多条规则。
- 规则之间支持优先级，默认按优先级从高到低执行。
- 每条规则需支持“命中即停止”或“命中后继续”两种策略。
- 当多条规则都可能修改同一字段时，以优先级和动作类型定义冲突处理顺序。

#### 5.4.4 节点模型需求

| 节点类型 | 作用 | 本期要求 |
| --- | --- | --- |
| 开始节点 | 定义触发场景、输入上下文 | P0 |
| 条件节点 | 比较订单字段、客户字段、商品字段、系统变量 | P0 |
| 分支节点 | 根据条件结果走不同路径，需支持默认分支 | P0 |
| 动作节点 | 修改状态、设置标签、写备注、锁单、指派仓库 | P0 |
| 结果节点 | 输出成功、人工处理、拦截等结果 | P0 |
| 计算节点 | 计算中间变量，如风险分、区域匹配结果 | P1 |
| 外部调用节点 | 调用白名单接口，如物流校验服务 | P2 |

条件能力要求：

- 支持等于、不等于、大于、小于、包含、为空、属于集合、时间范围比较。
- 支持多条件组合，至少支持 AND / OR。
- 支持引用订单主表字段、订单明细聚合结果、客户信息、系统时间、当前用户。

动作能力要求：

- 修改订单主状态或控制标记。
- 设置单值或多值标签。
- 写入系统备注或审核原因。
- 指定仓库或写入候选仓列表。
- 输出“转人工处理”结果。

#### 5.4.5 规则执行机制

- 规则执行以同步模式为主，要求在订单关键动作中可即时给出结果。
- 每次执行都要生成执行上下文快照，至少包含输入字段、命中的节点路径、输出结果和执行耗时。
- 规则执行结果必须能回写到订单，例如状态、标签、仓库建议、备注。
- 人工操作可以覆盖规则结果，但必须保留覆盖记录。
- 规则失败时不能导致订单无状态，需有明确降级策略，例如转人工审核或保留原状态。

#### 5.4.6 调试、发布与版本管理

- 规则支持草稿、已发布、已停用三种状态。
- 支持试运行：使用选定订单或样例数据模拟执行，查看节点命中路径。
- 支持版本保存：每次发布生成独立版本，不覆盖历史版本。
- 支持版本比对：至少能看到节点结构变化和关键参数变化。
- 支持回滚：可以将旧版本重新发布为当前生效版本。
- 订单详情页必须展示命中的规则版本号，便于解释历史处理结果。

#### 5.4.7 规则日志与可观测性

规则日志至少记录以下内容：

- 规则编码、规则名称、规则类型、版本号、执行场景。
- 触发来源，例如系统自动触发或人工重跑。
- 输入快照、命中条件、执行路径、输出结果。
- 执行状态、耗时、异常信息、重试记录。
- 与订单的关联关系，支持从订单详情反查规则日志。

配置人员应能基于日志回答三个问题：为什么这单被锁单、为什么分到了这个仓、为什么这次执行结果和上周不同。

#### 5.4.8 安全与边界约束

- 本期不支持执行任意 JavaScript、SQL 或脚本代码。
- 外部调用节点只能调用平台登记过的白名单接口。
- 规则输入上下文需经过脱敏与权限控制，避免不必要的敏感信息泄露。
- 停用规则后不得继续自动触发，但历史执行日志必须保留。

#### 5.4.9 模块验收标准

- 至少支持一条自动审核规则和一条自动分仓规则的可视化配置与发布。
- 规则试运行可以展示命中路径和输出结果。
- 规则执行结果能够回写订单并在详情页可追踪。
- 规则版本可回滚，历史订单可看到当时生效的规则版本。

### 5.5 用户与权限模块

- 支持管理员、运营、审核人员、配置人员四类默认角色。
- 角色需控制菜单权限、页面按钮权限、接口权限和配置发布权限。
- 关键动作必须纳入审计日志，如删除、规则发布、配置回滚、状态强制变更。
- 配置类页面应和业务操作页面分开授权，防止运营误改配置。
- 预留数据隔离能力，后续可扩展到商家级或组织级隔离。

### 5.6 日志、统计与运营支撑

- 系统日志：记录接口异常、运行异常、关键性能信息。
- 业务日志：记录订单状态变化、批量任务结果、人工处理说明。
- 规则日志：记录规则执行过程、版本、结果和异常。
- 审计日志：记录配置发布、字段删除尝试、权限变更、高风险业务动作。
- 基础统计指标：订单处理时长、待审核订单量、人工审核占比、规则命中率、分仓成功率、批量任务成功率。

## 6. 数据需求与核心模型

数据层必须同时支撑业务数据、配置数据和日志数据三类对象。第一类是订单主数据，第二类是低代码与规则元数据，第三类是执行过程留痕。

### 6.1 业务核心模型

| 表名 | 说明 | 核心字段 | 优先级 |
| --- | --- | --- | --- |
| users / roles / permissions | 用户、角色、权限 | id, name, code, status | P0 |
| orders | 订单主表 | order_no, source_no, status, customer_id, amount, warehouse_id | P0 |
| order_items | 订单商品明细 | order_id, sku_id, sku_name, qty, price | P0 |
| customers | 客户信息 | name, phone, level, tags | P0 |
| warehouses | 仓库信息 | code, name, region, priority, status | P0 |
| order_tags | 订单标签关系 | order_id, tag_code | P1 |
| shipments | 物流信息 | order_id, company_code, tracking_no, shipped_at | P1 |

### 6.2 配置与规则模型

| 表名 | 说明 | 核心字段 | 优先级 |
| --- | --- | --- | --- |
| entity_meta | 实体元数据 | entity_code, name, type, status | P0 |
| field_meta | 字段元数据 | entity_code, field_code, type, required, schema_json | P0 |
| dictionary_meta | 字典定义 | dict_code, item_code, item_name, sort | P0 |
| page_meta | 页面元数据 | page_code, entity_code, page_type, schema_json, version | P0 |
| action_meta | 页面动作元数据 | action_code, entity_code, scope, condition_json | P1 |
| rule_definitions | 规则定义 | rule_code, type, scene, status | P0 |
| rule_versions | 规则版本 | rule_id, version, graph_json, publish_info | P0 |
| import_templates | 导入模板 | entity_code, template_code, fields_json | P1 |

### 6.3 日志与审计模型

| 表名 | 说明 | 核心字段 | 优先级 |
| --- | --- | --- | --- |
| order_operation_logs | 订单操作日志 | order_id, operator_id, action, before_json, after_json | P0 |
| rule_exec_logs | 规则执行日志 | rule_version_id, scene, input_json, result_json, duration | P0 |
| audit_logs | 审计日志 | operator_id, action, target_type, target_id, detail_json | P0 |
| system_logs | 系统日志 | level, module, message, trace_id | P1 |

### 6.4 关键数据约束

- 核心业务表与配置表均需带创建时间、更新时间、创建人、更新人。
- 配置类表需要显式版本字段和状态字段。
- 订单扩展字段建议以结构化扩展表或 JSON 字段承载，但字段定义必须统一由 `field_meta` 管理。
- 规则执行日志应保存输入快照，避免规则变更后无法解释历史结果。
- 配置删除必须先校验依赖关系，禁止产生悬空引用。

## 7. 非功能需求

- 易用性：运营与配置人员应在经过简单培训后上手，关键页面需符合中后台使用习惯。
- 性能：在 1 万级订单数据规模下，常用列表查询应控制在 3 秒内；单次规则同步执行目标控制在 1 秒内。
- 稳定性：关键接口需具备参数校验、异常处理和错误留痕；批量任务不得静默失败。
- 安全性：登录鉴权、接口鉴权、操作审计、配置发布权限控制必须具备。
- 可扩展性：字段模型、页面模型和规则模型均应支持后续新增业务对象和节点类型。
- 可维护性：配置项、规则项、日志项需可追踪来源，避免“配出来但没人解释得清”的黑盒系统。
- 可部署性：系统必须支持部署到统一测试环境，便于验收和演示。

## 8. MVP 范围与阶段优先级

### 8.1 P0 必须完成

- 登录鉴权与角色权限。
- 订单列表、订单详情、审核、锁单、分仓、发货、取消。
- 订单日志、规则日志、审计日志。
- 实体、字段、页面的基础配置能力。
- 规则画布、规则保存、试运行、发布、回滚。
- 一套可部署的测试环境。

### 8.2 P1 建议完成

- 物流单号导入模板、导出模板与导入校验增强。
- 页面动作配置、视图级权限控制。
- 统计指标看板、基础运营概览。
- 仓配人员工作台。

### 8.3 P2 可延后

- 外部平台订单同步。
- 外部接口回调驱动的规则触发。
- 复杂消息通知、审批流、多租户 SaaS 化。

## 9. 验收标准

### 9.1 业务闭环验收

1. 至少完成一条订单从创建/导入到审核、分仓、发货、完成的完整链路。
2. 至少完成一条命中风险规则后转人工审核的异常链路。
3. 所有关键动作均可在订单详情中查看日志和操作人。

### 9.2 低代码能力验收

1. 新增至少一个订单扩展字段，无需改代码即可发布到页面。
2. 新增字段可出现在列表筛选、详情展示和规则条件中。
3. 页面配置支持预览、发布和回滚。

### 9.3 规则能力验收

1. 可通过可视化画布配置至少两类规则：自动审核、自动分仓。
2. 规则试运行可展示命中节点路径、输出结果与耗时。
3. 历史订单可查看命中的规则版本和执行日志。

### 9.4 工程验收

1. 系统可在测试环境中正常访问。
2. 关键错误可通过日志定位。
3. 需求、技术路线和实现范围保持一致，不出现明显超范围设计。

## 10. 后续扩展方向

以下能力不纳入本期，但本期设计应为其预留接口或扩展位：

- 外部电商平台订单同步、物流状态同步。
- 更细粒度的库存与仓配联动。
- 售后、退款、换货等逆向流程。
- 多组织、多商家、多租户隔离。
- 更丰富的规则节点类型，如异步回调、消息通知、外部审批。

## 11. 结论

本项目的第一阶段不是做一个“什么都能配”的平台，而是做一个围绕订单业务可落地的低代码系统。其核心价值不在于页面数量，而在于三件事是否同时成立：订单处理后台能跑通真实闭环；字段与页面调整不依赖开发改代码；规则判断可以用可视化方式沉淀并复用。只要这三个核心目标在 MVP 中成立，后续再扩展集成、复杂流程和多租户能力才是稳妥的。
