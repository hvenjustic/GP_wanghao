## 2. 当前开发主线

现阶段不要再回到“做 P0 基建”的节奏，接下来应围绕以下三条主线推进：

1. 订单效率主线：补齐导入导出、批量结果回显、异常处理联调和更完整的业务约束。
2. 低代码运行时主线：让发布后的实体/字段/页面配置真正驱动页面渲染，而不只停留在治理台。
3. 规则执行深化主线：补表达式库、多分支执行器和规则可解释性，让规则引擎更接近真实业务引擎。

## 4. 推荐执行顺序

建议按以下顺序推进，避免返工：

1. 先做任务包 A。
   这是最接近真实业务使用的短板，投入产出比最高。
2. 再做任务包 B。
   低代码平台现在治理能力已经具备，下一步应该补运行时，否则平台价值仍停留在后台配置。
3. 然后做任务包 C。
   规则引擎已能跑通，但缺少复杂逻辑和解释能力，适合在订单和低代码能力稳定后继续深化。
4. 最后做任务包 D。
   用户治理增强和运营支撑很重要，但优先级应低于订单效率和平台能力补齐。
