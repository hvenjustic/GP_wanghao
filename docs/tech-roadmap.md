# 面向中小电商的低代码订单管理系统 - 技术路线文档

> 从技术选型、系统架构到落地计划的实施文档

## 8. 技术路线总览

结合项目目标与“快速开发、快速上线”的要求，本项目推荐采用 TypeScript 全栈单仓模式，而不是从一开始采用重型微服务架构。这样做的主要原因是：前后端统一语言栈、项目结构更集中、接口联调成本更低、部署路径更短，更适合毕业设计/原型系统阶段。等系统跑通并验证需求后，再按需要拆分为独立后端服务。

| 技术层 | 推荐方案 | 选型原因 |
| --- | --- | --- |
| 前端与全栈框架 | Next.js (App Router) | 统一页面与接口开发，支持 Route Handlers，适合单仓全栈 |
| UI 组件 | Ant Design + ProComponents | 后台场景成熟，表格、表单、布局能力完善 |
| 数据库 | Supabase PostgreSQL | 托管 Postgres + Auth + Storage + Realtime，快速上线 |
| ORM | Prisma | 类型安全，迁移清晰，适配 Next.js 开发体验好 |
| 规则画布 | React Flow | 适合拖拽式节点编排与流程可视化 |
| 运行时校验 | Zod | 用于配置元数据和接口输入校验 |
| 部署 | Vercel + Supabase | 前端/Node 侧部署简单，数据库托管成本低 |

### 8.1 推荐技术架构

- 表示层：Next.js 页面、Ant Design 组件、ProTable/ProForm 组合。

- 配置层：实体元数据、字段元数据、页面元数据、规则图元数据。

- 业务层：订单服务、用户权限服务、规则执行服务、审计日志服务。

- 数据层：PostgreSQL 存储业务表与元数据表，Storage 存放附件。

- 平台能力：Supabase Auth 做认证，Row Level Security 为后续隔离预留能力。

### 8.2 为什么不建议当前阶段采用 Spring Boot + Vue 微服务

- 需要维护前后端两个主工程，接口联调与部署链路更长。

- 规则引擎与页面元数据本身就复杂，若再叠加微服务治理会显著拉长工期。

- 毕业设计更需要“尽快跑通业务闭环”，而不是过早引入架构复杂度。

- 后续如业务量增长，可平滑演进为 Next.js 前端 + NestJS 或 Spring Boot 独立后端。

## 9. 系统分层设计

| 层次 | 职责 | 关键产物 |
| --- | --- | --- |
| 前端界面层 | 页面展示、交互操作、配置画布 | 订单页面、配置页面、规则画布 |
| 渲染与解释层 | 解析元数据并生成页面和校验逻辑 | 表单 schema、列表 schema、详情 schema |
| 业务服务层 | 订单处理、规则执行、权限校验、日志记录 | 服务函数、Route Handlers |
| 数据持久层 | 业务数据、配置数据、执行日志持久化 | PostgreSQL 表、对象存储 |

## 10. 元数据与规则设计思路

本系统的核心不是单个页面，而是“以配置驱动页面与流程”。因此必须先设计元数据，再让渲染引擎和规则引擎去解释元数据。建议采用 JSON Schema 风格的结构来保存页面和字段配置，同时用结构化图数据保存规则画布。

### 10.1 页面元数据建议字段

- entityCode：关联的业务实体。

- fields：字段列表，包含字段编码、标题、类型、展示位置、校验规则。

- searchSchema：列表页搜索区域定义。

- tableSchema：列表列定义、格式化方式、排序能力。

- detailSchema：详情页卡片、时间线、子表定义。

### 10.2 规则元数据建议字段

- ruleCode / ruleType / version / status。

- nodes：节点数组，包含节点类型、标签、输入配置、输出配置。

- edges：连接关系数组。

- publishInfo：发布人、发布时间、版本备注。

- rollbackInfo：回滚来源、回滚原因。

## 11. 接口与模块划分建议

- 认证接口：登录、退出、获取当前用户、刷新会话。

- 订单接口：列表查询、详情读取、状态更新、发货、批量处理。

- 元数据接口：实体管理、字段管理、页面管理、发布配置。

- 规则接口：规则创建、保存、发布、停用、试运行、执行日志查询。

- 系统接口：字典、参数、日志、健康检查。

## 12. 项目目录结构建议（单仓）

```text
src/
app/                    # Next.js 页面与 Route Handlers
components/             # 通用组件
features/orders/        # 订单模块
features/meta/          # 元数据配置模块
features/rules/         # 规则画布与规则服务
lib/db/                 # Prisma client、数据库工具
lib/auth/               # 鉴权与会话处理
lib/schema/             # Zod schema 与渲染 schema
server/services/        # 业务服务层
prisma/                 # schema.prisma 与 migrations
docs/                   # 需求、接口、设计文档
```

## 13. 部署与环境方案

1. 开发环境：本地 Node.js + pnpm + Supabase 云数据库；使用 Prisma migration 管理表结构。

2. 测试环境：Vercel 部署 Web 与 Node 运行逻辑，Supabase 提供数据库、认证、对象存储。

3. 正式环境：保持与测试环境一致的最小差异部署方式，先确保可上线，再考虑自托管。

4. 日志与监控：至少保留应用日志、接口错误日志、规则执行日志和数据库备份策略。

| 周次 | 目标 | 交付物 | 优先级 |
| --- | --- | --- | --- |

## 15. MVP 范围定义

MVP 的目标不是覆盖全部电商场景，而是确保系统形成一个完整闭环：用户可登录系统，查看订单，完成审核与发货；实施人员可定义订单实体字段与页面；规则人员可拖拽配置一条审核或分仓规则；系统可以保存配置、执行规则并记录结果；最后系统可以部署到测试环境供他人试用。

- 必须完成：登录鉴权、订单管理、元数据配置、规则画布、部署上线。

- 建议完成：导入导出、附件上传、简单统计面板、规则试运行。

- 可以延期：复杂集成、多租户账单、复杂审批流、消息中心。

## 16. 验收标准

1. 业务闭环可运行：至少完成一个完整订单从进入系统到审核、分仓、发货的流程。

2. 低代码能力可验证：通过配置新增至少一个扩展字段，并在页面中自动呈现。

3. 规则能力可验证：通过可视化配置完成至少一条自动审核或分仓规则并成功执行。

4. 可部署：系统可在测试环境中正常访问。

5. 可验证：能够采集日志并输出关键指标，用于后续案例研究。

## 17. 风险与应对

- 风险一：规则引擎范围失控。应对：本期只做基础节点与同步规则，不引入复杂脚本执行。

- 风险二：低代码配置过于抽象导致难落地。应对：先围绕订单实体做定制型低代码，而非试图做通用平台。

- 风险三：数据库结构频繁变化。应对：用 Prisma migration 管理表结构，配置数据尽量以 JSON 形式承载。

- 风险四：上线时间紧。应对：严格执行 P0/P1/P2 优先级，确保先完成核心闭环。

## 18. 最终结论

本项目的最佳开工策略是：以订单管理为核心业务对象，以元数据驱动和规则编排为核心技术特色，以 Next.js 全栈单仓 + Supabase/PostgreSQL 为基础设施，以 6 周左右完成可用 MVP 为第一阶段目标。这样既保留了开题报告中“领域建模、动态渲染、规则引擎、案例验证”的研究主线，又显著降低了原型开发的工程复杂度，更适合快速开发与快速上线。

## 附录 A. 开发阶段的关键决策清单

- 决策 1：主技术栈采用 Next.js，不采用当前阶段的 Spring Boot + Vue 微服务。

- 决策 2：数据库采用 PostgreSQL，并同时承载业务数据与配置数据。

- 决策 3：规则引擎先做可视化同步规则，不做复杂异步编排。

- 决策 4：低代码能力先服务订单场景，不追求一开始通用于全部业务。

- 决策 5：上线环境优先选择托管平台，减少运维成本。
